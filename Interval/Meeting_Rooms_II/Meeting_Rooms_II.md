### 1. 核心題意與挑戰

給定一個會議時間安排的陣列 `intervals`，每個會議時間會包括開始和結束的時間 `[[s1,e1],[s2,e2],...] (si < ei)`。
請你計算並返回「至少需要多少個會議室」，才能滿足這些會議同時進行，且不互相衝突。

* **隱藏挑戰**：
  * 這題與上一題 Merge Intervals 有極大的不同。我們不是要合併把它變少，我們是想要知道「同一時間**最多**有多少個區間交疊在一起」。
  * 這被譽為「LeetCode 區間題的至尊經典」，面試出現頻率極高。

---

### 2. 解法對比與完整程式碼

#### A. 最小堆積 (Min-Heap / Priority Queue) —— **最符合生活經驗的解法**

**思路**：
想像你是一個會議室調度員。
先把所有預定**按照開始時間排序**。為什麼？因為你一定是照著時間順序來安排會議室。
接著，我們需要知道目前「哪一間會議室最快結束放空」？所以我們維護一個最小堆疊 (Min-Heap)，裡頭放的只有各個會議室的**「結束時間」**。
* 每當有一個新會議要開始時，看看堆疊頂端（也就是目前最早會結束的那場會議）：
  * 如果 `頂端結束時間 <= 新會議開始時間`，代表那間會議室空出來了！我們把它彈出 (Pop)，並把新會議的結束時間加進去 (Push)。這意味著我們重複利用了同一個房間。
  * 如果 `頂端結束時間 > 新會議開始時間`，代表目前所有的房間都還在開會，我們只能無奈地開一間新會議室，把新結束時間加進去 (Push)。
* 最後，Heap 裡面有幾個結束時間，我們就開了幾間房間。

* **時間複雜度**：$O(N \log N)$ (排序 $O(N \log N)$ + 遍歷每個會議維護堆積 $O(N \log N)$)
* **空間複雜度**：$O(N)$ (Heap 的空間)

```python
import heapq
from typing import List

class Solution:
    def minMeetingRooms(self, intervals: List[List[int]]) -> int:
        if not intervals:
            return 0
            
        # 1. 依照會議開始時間從小到大排序
        intervals.sort(key=lambda x: x[0])
        
        # 2. 準備一個 Min-Heap 存放各個會議室目前的「結束時間」
        free_rooms = []
        
        # 把最早開始的會議的結束時間丟進去 (也就是開了第一間房)
        heapq.heappush(free_rooms, intervals[0][1])
        
        # 3. 開始調度後面的每一場會議
        for meeting in intervals[1:]:
            # 最早開完的會議室空出來了嗎？
            # free_rooms[0] 是 heap 中最小的值 (即最早結束時間)
            if free_rooms[0] <= meeting[0]:
                # 房間空出來了，舊的結束時間可以刪掉了，這間房交給新會議用
                heapq.heappop(free_rooms)
                
            # 不論有沒有空出來，這個新會議最終都需要一間房記錄它的結束時間
            heapq.heappush(free_rooms, meeting[1])
            
        # Heap 裡有幾個時間，就代表有幾個彼此重疊而分開的房間
        return len(free_rooms)
```

#### B. 時間軸座標拆解法 (Chronological Sweeping Line) —— **非常神妙的巧解**

**思路**：
不要把它想成一個個「區間」。
把所有會議的 `start` 和 `end` 各自抽出來，變成兩個獨立的陣列，分別排序。
我們用兩個指針 `s_ptr` 和 `e_ptr` 分別指向開始時間和結束時間的頭。
如果 `start[s_ptr] < end[e_ptr]`，代表有新的會議開始了，但還沒有會議結束。所以我們 `rooms += 1`，並把 `s_ptr` 推進。
如果有會議結束，代表空出房間了 `rooms -= 1`，`e_ptr` 推進。
歷史上 `rooms` 曾達到的最高點，就是我們需要的會議室總數。這就好像我們在時間軸上掃描，遇到 Start 人數+1，遇到 End 人數-1，求最高人數。

* **時間複雜度**：$O(N \log N)$ (依然要雙重排序)
* **優點**：邏輯漂亮，但對部分人來說不如 Heap 直觀。

### 3. 實務應用場景

這其實是**最大圖著色問題 (Graph Coloring)** 在區間圖 (Interval Graph) 上的應用。

#### 1. 雲端運算資源動態調度 (EC2 Auto Scaling)
* **應用**：每個客戶提交了運算 Job 的開始預估與結束預估時間。AWS 想要知道「在這段時間內，我最多需要同時啟動幾台虛機節點才夠大家用？」底層就是這套 Min-Heap 機制。

#### 2. 尖峰流量併發數預估 (Concurrency Analysis)
* **應用**：將使用者的連線登入與登出日誌，轉化成事件的開始與結束點。這題的 Sweep Line 解法就是在畫出監控系統上的 Active Concurrent Users 曲線圖。
