### 1. 核心題意與挑戰

公路上有一群車子，每台車有它現在的位置 `position[i]` 和它的速度 `speed[i]`。所有的車子都是往著同一個目的地 `target` 前進。
公路只有單線道，車子不能超車。
當後面開得快的車子追上前面開得慢的車子時，它就**必須減速，跟著前面的車子組成一個車隊 (Car Fleet)**，以同樣的速度前進。
一輛車就算是自己開，也算是一個車隊。
請問當所有車子到達目的地 `target` 時，總共會有幾個車隊？

* **隱藏挑戰**：
  * 面試中，遇到這種追擊問題，直覺是用數學公式去聯立求解，或是用微積分去模擬每秒鐘。但這不可能在程式中用 $O(N)$ 漂亮處理完。
  * 追撞的現象，本質上是一種「**前面的擋住了後面的**」的後進先出機制，這就是 Stack 題。

---

### 2. 解法對比與完整程式碼

#### 單調堆疊 (Monotonic Stack) —— **把時間化為空間的解法**

**思路**：
如果車子 A 在車子 B 之前，車子 B 要追上 A，唯一的條件是：**B 到達終點所需的時間小於或等於 A 到達終點所需的時間**。
這句話是整題的精髓！我們不需要去算他們在哪一秒、哪一公里相撞。只要知道他們「單獨跑的話誰先到終點」就好了。

1. **綑綁並排序**：先將每一台車的 `(位置, 速度)` 綁定在一起，然後**根據位置由大到小排序**（也就是從最靠近終點的那台車開始看）。
2. **計算到達時間**：由於是從最靠近終點的車開始看，我們維護一個 Stack。我們把每台車的達到時間 $\frac{Target - Position}{Speed}$ 算出來。
3. **判斷追撞**：如果目前這台車算出來的時間 $\le$ Stack 頂端的車（前面那台車），代表它一定會在到達終點前追撞上去，成為同一個車隊。既然成為同一個車隊，我們就假裝它不存在了（因為速度被前面的人限制死，它的到達時間等同於前車），不把它推入 Stack。
4. **形成新車隊**：如果算出來的時間 $>$ Stack 頂端的車，代表它太慢了，追不上！那這台車自己就會成為一個新車隊的頭，把它推入 Stack 中。
5. 最後 Stack 裡面有幾個時間紀錄，就代表有幾個車隊。

* **時間複雜度**：$O(N \log N)$ (排序的成本，Stack 操作僅 $O(N)$)
* **空間複雜度**：$O(N)$

```python
from typing import List

class Solution:
    def carFleet(self, target: int, position: List[int], speed: List[int]) -> int:
        if not position:
            return 0
            
        # 將 (位置, 速度) 打包在一起，並根據位置降序排列 (從離終點最近的開始看)
        cars = sorted(zip(position, speed), reverse=True)
        
        stack = []
        
        for pos, spd in cars:
            # 計算如果這台車自己開，到達終點需要的時間 (浮點數)
            duration = (target - pos) / spd
            
            # 判斷是否會追撞
            if not stack:
                stack.append(duration)
            else:
                # 當前車到達的時間如果 <= 頂端那台車的時間
                # 代表一定會在終點前追撞上！所以它會被融入前車的車隊中，就不特別把它放進去了。
                # 如果 duration > stack[-1]，代表追不上，它會自己形成新車隊的頭，推入 stack。
                if duration > stack[-1]:
                    stack.append(duration)
                    
        return len(stack)
```

### 3. 實務應用場景

#### 1. 網路路由器中的封包壅塞控制 (Congestion Control / Traffic Shaping)
* **應用**：在 Token Bucket 或 Leaky Bucket 等流量整形演算法中，如果上游發送的高速封包遇到了下游頻寬較小的瓶頸節點，封包就會產生類似 Car Fleet 的積壓效應。透過計算到達時間，可以預測在 Router 緩衝區需要開多大的 Queue 才能承載這個 Fleet 而不丟包。

#### 2. GPU 渲染管線 (Graphics Pipeline) 任務聚合
* **應用**：Shader 處理時如果後面的輕量運算任務「追上」了前面佔用大量時間的材質渲染任務，會被迫在此產生 Pipeline Stall。這與 Car Fleet 的邏輯完全相似。

### 4. 總結筆記

| | 說明 |
| --- | --- |
| **Why Sort Reverse?** | 從最靠近終點的車開始看，因為它不會被任何人擋住（它是自由的）。只有把它的真實到達時間算出來了，才能作為攔截後面車輛的基準點。 |
| **Why `float`?** | 因為就算是在前一公里追上，也是追上。不能使用整數除法 `//`，否則時間的微小差異會被抹去導致誤判車隊。 |
